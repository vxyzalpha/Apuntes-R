Sección 01

¿Qué es el scope?

El scope o alcance define desde dónde es visible y accesible una variable. Entenderlo es la diferencia entre código que funciona y código que falla misteriosamente.

Idea central

En R, cada llamada a una función crea su propio espacio de variables. Las variables que creas dentro de una función no existen fuera de ella, y las de fuera son visibles dentro… salvo que la función tenga la suya propia con el mismo nombre.

El problema que resuelve

Sin scope, todos los programas compartirían las mismas variables. Cualquier función podría modificar accidentalmente datos de otra. El scope es el mecanismo que evita esas colisiones.

intro_01.REjecutable
rm(list = ls())  # limpia el workspace

x <- 100         # x en el entorno GLOBAL

f <- function() {
  x <- 42         # x LOCAL: solo existe dentro de f()
  cat("Dentro de f, x =", x, "\n")
}

f()
cat("Fuera de f,   x =", x, "\n")  # sigue siendo 100
salida

Observa que la x local de f() y la x global son completamente independientes. Modificar una no afecta a la otra.

¿Por qué es importante aprenderlo?

  • Evitar bugs silenciosos donde una variable cambia sin que lo esperes.
  • Entender por qué R dice "object not found" en ciertas situaciones.
  • Usar closures correctamente.
  • Saber cuándo usar <<- y cuándo <-.
Nota sobre rm(list = ls())

Todos los bloques de código de estas páginas empiezan con rm(list = ls()). Esto limpia el workspace antes de cada ejemplo para que los conceptos se vean con claridad, sin interferencias de variables de bloques anteriores.

Sección 02

Un entorno es el contenedor donde R guarda variables. Cada función crea uno nuevo al ejecutarse. Los entornos se organizan en cadena, cada uno apuntando a su padre.

Sección 03

Scoping léxico

R usa lexical scoping: una función busca sus variables en el entorno donde fue definida, no en el entorno desde el que se la llama.

Regla de oro

Cuando R necesita el valor de una variable libre (no es argumento ni local), la busca en el entorno de definición de la función, no en el entorno de llamada.

Variable libre vs. local

Una variable libre es aquella que se usa dentro de una función pero no está declarada localmente (ni como argumento ni con <- en el cuerpo).

lexical_01.REjecutable
rm(list = ls())

x <- "GLOBAL"

f <- function() {
  cat("f ve x =", x, "\n")  # x es libre: busca donde f fue DEFINIDA
}

g <- function() {
  x <- "LOCAL de g"
  f()   # f se llama desde g, pero f NO ve la x de g
}

f()   # busca x en global → "GLOBAL"
g()   # f sigue viendo la x global, no la de g
salida
Léxico vs. dinámico

Si R usara dynamic scoping (como bash), f() dentro de g() vería "LOCAL de g". Pero R es léxico: el entorno que importa es donde se escribió la función.

¿Cuándo se resuelve la variable libre?

lexical_02.REjecutable
rm(list = ls())

# La variable libre se busca en TIEMPO DE EJECUCIÓN,
# pero en el entorno de DEFINICIÓN.

base_val <- 10

sumar_base <- function(n) n + base_val  # base_val es libre

sumar_base(5)    # 15: base_val = 10

base_val <- 100   # cambiamos la variable global

sumar_base(5)    # 105: ahora base_val = 100
salida
Cuidado

El comportamiento de sumar_base() cambia según el valor de base_val en el momento de la llamada. Las funciones con variables libres son menos predecibles. Cuando puedas, pasa todos los datos como argumentos.

¿Qué imprime g() en este código?
x <- 1; f <- function() x; g <- function() { x <- 2; f() }; g()
A2 — porque g tiene x = 2 y llama a f()
B1 — f fue definida en el global donde x = 1 (scoping léxico)
CError — x no está definida dentro de f()

Enmascaramiento (shadowing)

Una variable local con el mismo nombre que una de un entorno padre la enmascara: R encuentra la local primero y nunca llega a buscar más arriba.

busqueda_02.REjecutable
rm(list = ls())

valor <- "soy global"

externa <- function() {
  valor <- "soy de externa"   # enmascara la global

  interna <- function() {
    valor <- "soy de interna"  # enmascara la de externa
    cat("interna ve:", valor, "\n")
  }

  interna()
  cat("externa ve:", valor, "\n")
}

externa()
cat("global  ve:", valor, "\n")
salida
Buena práctica

Evita usar el mismo nombre de variable en distintos niveles de scope. Aunque R lo permite, hace el código difícil de razonar. Si necesitas el valor del nivel superior, pásalo como argumento.

Sección 05

El operador <<-

<<- asigna en el primer entorno padre donde la variable ya exista. Si no la encuentra en ningún padre, la crea en el global. Es poderoso, pero debe usarse con criterio.

OperadorDónde asignaCuándo usar
<-Siempre en el entorno local actualSiempre por defecto
<<-Busca hacia arriba; asigna donde encuentre la variable (o en global si no existe)Closures con estado mutable

El bug clásico con <-

super_01.REjecutable
rm(list = ls())

total <- 0

# ❌ <- crea una variable LOCAL: el global no cambia
acumular_mal <- function(x) {
  total <- total + x  # local, no modifica el global
}

acumular_mal(10); acumular_mal(20)
cat("total (mal):", total, "\n")   # sigue siendo 0 !

# ✅ <<- sube al entorno padre y modifica la variable allí
acumular_bien <- function(x) {
  total <<- total + x
}

acumular_bien(10); acumular_bien(20)
cat("total (bien):", total, "\n")  # 30
salida

<<- en funciones anidadas

super_02.REjecutable
rm(list = ls())

# <<- sube solo hasta donde encuentra la variable
# NO necesariamente llega al global

externa <- function() {
  x <- "soy de externa"

  interna <- function() {
    x <<- "modificada por interna"
    # sube un nivel: encuentra x en 'externa', la modifica allí
  }

  interna()
  cat("x en externa:", x, "\n")
}

externa()
cat("x existe en global?:", exists("x"), "\n")  # FALSE
salida
Cuándo evitar <<-

Usar <<- introduce estado mutable global, lo que dificulta razonar sobre el código y puede causar bugs difíciles de reproducir. La alternativa más limpia casi siempre es devolver el valor con return() y reasignarlo desde fuera.

x <- 5; f <- function() { x <<- x * 2 }; f(); cat(x) ¿qué imprime?
A5 — <<- no funciona dentro de funciones
B10 — <<- modifica la x del global
CError — x no está definida en f()
Sección 06

Closures

Un closure es una función que recuerda el entorno en el que fue creada. En R, todas las funciones son closures.

Definición

Toda función en R lleva una referencia al entorno donde fue definida. Cuando ese entorno contiene datos que la función usa, decimos que la función cierra sobre (closes over) esos datos. De ahí el nombre closure.

Fábrica de funciones

closure_01.REjecutable
rm(list = ls())

multiplicador <- function(factor) {
  function(x) x * factor   # 'factor' vive en el closure
}

doble  <- multiplicador(2)
triple <- multiplicador(3)

doble(10)    # 20: su closure recuerda factor = 2
triple(10)   # 30: su closure recuerda factor = 3

# Cada una tiene su propio entorno independiente
environment(doble)$factor
environment(triple)$factor
salida

Closure con estado mutable

Combinando closures con <<- podemos encapsular estado, sin exponerlo al entorno global.

closure_02.REjecutable
rm(list = ls())

hacer_contador <- function(inicio = 0) {
  n <- inicio   # estado encapsulado en el closure

  list(
    sumar  = function(x = 1) { n <<- n + x; invisible(n) },
    restar = function(x = 1) { n <<- n - x; invisible(n) },
    ver    = function()        n,
    reset  = function()        { n <<- inicio; invisible(n) }
  )
}

c1 <- hacer_contador(10)
c2 <- hacer_contador(0)

c1$sumar(5); c1$sumar(3)
c1$ver()    # 18

c2$sumar(100)
c2$ver()    # 100 — independiente de c1

c1$reset(); c1$ver()  # 10 — vuelve al inicio
salida
Cuándo usar closures

Son ideales para: fábricas de funciones, estado encapsulado sin clases formales, memoización, y callbacks que necesitan recordar contexto.

make_adder <- function(n) function(x) x + n; add10 <- make_adder(10); n <- 999; add10(3) ¿qué devuelve?
A1002 — usa la n global (999 + 3)
B13 — el closure recuerda n = 10 de su entorno de definición
CError — n está redefinida
Sección 07

Errores frecuentes

El 90 % de los errores relacionados con el scope siguen los mismos tres patrones. Reconocerlos te ahorra horas.

Error 1 — usar <- cuando se quería <<-

err_01.REjecutable
rm(list = ls())

log_events <- c()

# ❌ <- crea log_events LOCAL en cada llamada, el global no cambia
registrar_mal <- function(ev) {
  log_events <- c(log_events, ev)
}

registrar_mal("inicio")
registrar_mal("proceso")
cat("log (mal):", log_events, "\n")  # vacío!

# ✅ <<- modifica el global
registrar_bien <- function(ev) {
  log_events <<- c(log_events, ev)
}

registrar_bien("inicio")
registrar_bien("proceso")
cat("log (bien):", log_events, "\n")
salida

Error 2 — asumir que una variable local existe fuera

err_02.REjecutable
rm(list = ls())

preparar <- function() {
  resultado <- 42   # solo vive dentro de preparar()
}

preparar()

# ❌ resultado no existe en el global
tryCatch(
  cat(resultado),
  error = function(e) cat("❌", e$message, "\n")
)

# ✅ fix: devolver el valor con return implícito
preparar2 <- function() {
  resultado <- 42
  resultado          # última expresión = retorno
}
resultado <- preparar2()
cat("✅ resultado =", resultado, "\n")
salida

Error 3 — closures en loops con variable compartida

err_03.REjecutable
rm(list = ls())

# ❌ todas las funciones ven el mismo i global (el del final)
fns <- vector("list", 3)
for (i in 1:3) {
  fns[[i]] <- function() i   # i es libre, apunta al global
}
cat("Bug:  ", fns[[1]](), fns[[2]](), fns[[3]](), "\n")

# ✅ local() crea un entorno nuevo por iteración, captura i
for (i in 1:3) {
  fns[[i]] <- local({
    j <- i          # congela el valor actual
    function() j
  })
}
cat("Bien: ", fns[[1]](), fns[[2]](), fns[[3]](), "\n")
salida
Por qué ocurre el bug del loop

En el for-loop, i es una variable global que se actualiza en cada iteración. Todas las funciones creadas apuntan a la misma i. Al terminar el loop, i = 3 para todas. local() crea un entorno nuevo en cada iteración, por lo que cada j queda "congelada" con su valor.

Ejercicio 1 — ¿Qué vale x?

Sin ejecutar, predice qué imprime cada cat(). Luego ejecuta para comprobarlo.

ej01.REjecutable
rm(list = ls())

x <- 1

f <- function() {
  x <- 2
  g <- function() {
    x <- 3
    cat("g ve x =", x, "\n")   # ¿cuánto?
  }
  g()
  cat("f ve x =", x, "\n")     # ¿cuánto?
}

f()
cat("global x =", x, "\n")     # ¿cuánto?
salida
▸ Explicación
Respuesta

Cada función tiene su propia x local: g ve 3, f ve 2, global ve 1. Ninguna modifica las demás porque todas usan <- (local). Tres entornos distintos, tres valores distintos.

ej02_sol.RSolución
rm(list = ls())

hacer_acumulador <- function() {
  total <- 0
  function(x) {
    total <<- total + x   # modifica total en el closure
    total
  }
}

acc  <- hacer_acumulador()
acc2 <- hacer_acumulador()   # independiente de acc

acc(10); acc(5); acc(3)   # 10, 15, 18
acc2(100)                  # 100 — no interfiere
